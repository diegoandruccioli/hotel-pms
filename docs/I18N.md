# Hotel PMS — Internazionalizzazione (i18n)

**Stack:** `react-i18next` + `i18next` — file JSON puri, senza `.i18n.ts` co-locati  
**Lingue supportate:** `en` (inglese, default) · `it` (italiano)  
**Versione:** 1.0 — 2026-05-07

---

## 1. Struttura dei file

```
frontend/src/locales/
├── en/                     ← lingua di default e fallback
│   ├── admin.json          ← gestione utenti, profilo hotel, Alloggiati auto-send
│   ├── auth.json           ← login, registrazione, cambio password, errori auth
│   ├── billing.json        ← fatture, pagamenti, metodi di pagamento
│   ├── calendar.json       ← calendario, planning board, trascinamento
│   ├── common.json         ← tutto il resto (nav, dashboard, camere, ospiti, prenotazioni, UI globale)
│   ├── dashboard.json      ← statistiche dashboard
│   ├── errors.json         ← messaggi di errore generici e validazione
│   ├── guests.json         ← anagrafica ospiti, documenti identità
│   ├── housekeeping.json   ← status pulizia camere
│   ├── reservations.json   ← form prenotazione, selezione camere
│   ├── restaurant.json     ← ordini F&B, menu, conferma
│   ├── rooms.json          ← camere fisiche, tipologie
│   ├── settings.json       ← impostazioni UI (tema, lingua)
│   └── stays.json          ← check-in, check-out, walk-in, Alloggiati PS
└── it/                     ← traduzione italiana (stessa struttura)
    └── [14 file identici per nome]
```

**Regola:** ogni chiave presente in `en/` deve essere presente anche in `it/`. Il fallback è sempre `en` se una chiave manca in `it`.

## 2. Configurazione i18next

Il punto di ingresso è `src/i18n.ts`. Configurazione principale:

```typescript
i18n
  .use(initReactI18next)
  .init({
    resources,           // namespace caricati staticamente (bundle Vite)
    lng: 'en',           // lingua iniziale
    fallbackLng: 'en',   // fallback se chiave mancante
    defaultNS: 'common', // namespace di default se non specificato
    ns: ['common', 'auth', 'stays', 'billing', 'guests', 'reservations',
          'rooms', 'restaurant', 'housekeeping', 'calendar', 'admin',
          'dashboard', 'errors', 'settings'],
    interpolation: { escapeValue: false },
  });
```

La lingua viene cambiata a runtime tramite `i18n.changeLanguage('it')` nello store `settingsStore`.

## 3. Uso nei componenti

### Namespace singolo (il più comune)

```typescript
import { useTranslation } from 'react-i18next';

const { t } = useTranslation('stays');
// → usa il namespace 'stays' (en/stays.json o it/stays.json)

t('check_in_title')           // → "Check-In"
t('walkin_loading_rooms')     // → "Loading available rooms..."
```

### Namespace multipli

Usato quando un componente ha bisogno di chiavi da più namespace:

```typescript
const { t } = useTranslation(['stays', 'common']);

t('stays:check_in_title')    // → cerca in stays.json
t('common:cancel')           // → cerca in common.json
t('cancel')                  // → cerca in namespace principale (stays), poi fallback a common
```

### Interpolazione variabili

```typescript
t('common:alloggiati_report_downloaded', { date: '2026-05-01' })
// en: "Alloggiati report for 2026-05-01 downloaded."
// it: "Report Alloggiati per 2026-05-01 scaricato."
```

Nei file JSON: `"alloggiati_report_downloaded": "Alloggiati report for {{date}} downloaded."`

## 4. Convenzioni per le chiavi

### Prefissi funzionali

| Prefisso | Significato | Esempi |
|----------|------------|--------|
| `nav_` | Voci di navigazione | `nav_dashboard`, `nav_stays`, `nav_rooms` |
| `label_` | Label di form o campo | `label_checkin_date`, `label_checkout_date` |
| `err_` | Messaggi di errore specifici | `err_guest_not_found`, `err_stay_id_required` |
| `msg_` | Messaggi informativi | `msg_select_guest`, `msg_select_room` |
| `action_` | Azioni (bottoni, link) | `action_checkout`, `action_confirm` |
| `hint_` | Testi di aiuto contestuali | `hint_alloggiati_auto_send` |
| `tab_` | Tab di navigazione interna | `tab_rooms`, `tab_room_types` |
| `stat_` | Statistiche della dashboard | `stat_total_guests`, `stat_active_stays` |
| `status_` | Label di stato | `status_confirmed`, `status_checked_in` |
| `role_` | Label dei ruoli utente | `role_admin`, `role_receptionist` |
| `payment_method_` | Metodi di pagamento | `payment_method_cash`, `payment_method_card` |

### Chiavi senza prefisso

Le chiavi comuni e brevi vivono direttamente senza prefisso: `save`, `cancel`, `edit`, `delete`, `close`, `confirm`, `back`, `name`, `email`, `status`, `date`, `amount`, `loading`.

### Formato

- **snake_case** per tutte le chiavi
- **Minuscolo** (mai UPPER_CASE)
- **Nessun namespace nel nome della chiave** — il namespace è il file JSON

## 5. Come aggiungere una nuova stringa

1. Identificare il namespace corretto in base alla feature  
2. Aggiungere la chiave in `en/<namespace>.json`  
3. Aggiungere la traduzione in `it/<namespace>.json` (stessa chiave, testo tradotto)  
4. Usare `t('chiave')` nel componente con il namespace corretto  
5. Se la stringa usa variabili, usare `{{nomeVar}}` nel JSON e passare l'oggetto a `t()`

**Esempio:**

```jsonc
// en/billing.json
"invoice_closed_success": "Invoice {{number}} closed successfully."

// it/billing.json
"invoice_closed_success": "Fattura {{number}} chiusa con successo."
```

```typescript
t('billing:invoice_closed_success', { number: 'INV-042' })
```

## 6. Regola anti-hardcoding

**Zero testo visibile all'utente fuori da i18n.** Il linter e il code review devono bloccare:

```tsx
// ❌ VIETATO
<button>Save</button>
<p>Error loading data</p>

// ✅ CORRETTO
<button>{t('common:save')}</button>
<p>{t('common:failed_load_data')}</p>
```

Unica eccezione ammessa: testo puramente tecnico non visibile all'utente (attributi `aria-label` già forniti via `t()`, log console, codici interni).

## 7. Fallback e chiavi mancanti

- Se una chiave manca in `it/`, viene usata la versione `en/` senza errore visibile
- Se una chiave manca in entrambi, `react-i18next` mostra la chiave stessa (es. `"missing_key"`) — utile per identificare chiavi dimenticate in sviluppo
- In test (Vitest), il mock standard è:
  ```typescript
  vi.mock('react-i18next', () => ({
    useTranslation: () => ({ t: (key: string) => key, i18n: { language: 'en' } }),
  }));
  ```
  Questo fa sì che `t('nav_rooms')` ritorni la stringa `"nav_rooms"` — i test verificano la chiave, non il testo tradotto.
